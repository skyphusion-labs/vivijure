# v1.0 Output-Verified Coverage Checklist

**Living release-gate doc.** Every render path and user-facing feature, graded by the ONLY
standard that counts for a render feature: **did a human look at the actual output artifact?**
`phase=done` / "it completed" / conformance-green is necessary but NOT sufficient. A line is
`VERIFIED` only with a linked, inspected artifact (ffprobe nb_streams AND eyeballed pixels, not
noise/garbage). No linked proof -> it is `UNTESTED`, full stop.

This doc exists because the S40 "total shakedown" reported "we tested every single option" when it
had verified RUN + WIRE, not OUTPUT. Two things slipped through: the CogVideoX-16gb door (never
produced a watchable frame) and scatter/gather (0 recent e2e runs). Trust is rebuilt one linked
artifact at a time. Do not mark anything green you have not looked at.

## Grades

- **VERIFIED** -- artifact inspected (ffprobe + eyeball), proof linked, this release train.
- **STALE** -- was genuinely verified, but weeks ago / on an older build; needs a re-run to trust.
- **BROKEN** -- output inspected and confirmed garbage/wrong.
- **UNTESTED** -- no inspected-artifact proof exists. The honest default.

Last updated: 2026-07-13 (Mackaye). **Live shakedown in progress** -- driving real cloud-GPU renders
and eyeballing every artifact. Test bundle: `bundles/dawn_balloon-f132895466db8a5d.tar.gz` (2-shot,
no cast). Frames archived in the session scratchpad.

## Findings (new, this shakedown)

- **F1 -- cross-render clip-adoption contamination (concurrent, same project).** Running film A
  (seedance), film B (kling), and a scatter (minimax) off the SAME bundle meant all three derived the
  SAME project namespace + same shot_ids. The scatter's gathered output came back **byte-identical**
  (md5 `78501bfb...`) to film A's seedance film -- confirmed NOT an artifact-cache artifact (film B
  returns distinct bytes via the same route; cache-bust unchanged). The clip-adoption/reuse feature
  (#583: "reuse a prior same-project render's R2 artifact") appears to adopt ANOTHER render's clips when
  two renders share project+shot_id, ignoring that they used different motion backends. Real usage gives
  each project a distinct namespace, so the blast radius is narrow, BUT: (a) adoption keyed on
  project+shot_id without a backend/config fingerprint can serve the wrong bytes; (b) it silently
  produced a "COMPLETED" scatter that was really a different film. Needs a real fix or a fingerprint
  guard before v1.0. Isolated re-run (`scatter-9c2dbf99`, unique project) is the clean scatter verdict. **FIXED: PR #768**
(rollins, green, not self-merged) -- provenance `.prov` sidecar fingerprints clip reclaim (backend/config/
keyframe/prompt); adopts only on identical-config match else re-renders; +9 regression tests; issue #767.

- **F2 -- audio-master ships a SILENT film (BROKEN). ROOT-CAUSED (4-link chain):** (1) music-gen ignores
  `seconds` -> 381s bed (F3); (2) `containers/audio-master` masters the FULL bed to a ~73MB untrimmed wav
  (no trim to film length); (3) `containers/video-finish` `_download` caps audio at 64MB (`app.py:35`) ->
  silently drops the over-cap bed, returns `ok:true, hasAudio:false`; (4) the studio ignored `hasAudio` ->
  green render, no audio = the silent-degrade violation. **FIX (two parts):** studio honesty = **PR #765**
  (rollins, MERGED-PENDING): `enterMuxPhase`/`muxAudioOntoVideoKey` now take the observable degrade /
  return 422 on `hasAudio===false`, never a silent green (+3 regression tests, 1638 pass). Audibility =
  **container fix IN PROGRESS** (strummer): trim the mastered bed to film length so it survives the cap;
  rebuild + deploy the prod media swarm image. Not DONE until an e2e re-run shows `nb_streams=2`.
- **F3 -- music-gen ignores the `seconds` length param.** `/api/storyboard/score-bed` with `seconds:12`
  returned a 381s (6.4 min) track. Not honored end-to-end. Minor on its own, but it's the likely trigger
  for F2. Fix: honor `seconds` (trim/generate to length) or document it as advisory + make the mux trim.
